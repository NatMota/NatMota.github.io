API design and remote working productivity

API design and remote working productivity

In this article I've set out to explore how API's might be similar to remote work, and if there are any rules from API design that can help us be more productive people. I didn't know where I would land before starting to write about it, but I'm happy with where this exploration got me. I wrote my learnings at the end.

Going one layer up

Remote work mirrors in-office work fairly well, but communication is one aspect that loses nuance. Whatever doesn’t fit in the data cable must stay behind, for better or for worse. Since smoke signals, you can reach far but there is only so much you can say.

Machines need to do remote work almost all of the time, often with other systems which are complete strangers, without using tone of voice, body language, or facial expressions.

Are there any lessons we can take from how machines communicate? Maybe the most important one that we can apply to remote working is to be as intentional as you can.

What are APIs?

APIs are one way computers communicate. They stand for Application Programming Interfaces. You can use APIs yourself to talk with a computer, but more importantly to get many computers to work with each other. Jeff Lawson, former Twilio CEO, has written a very approachable book introducing them.

He uses banking as an example in the book. “Nowadays your bank is a mobile app and you like your bank if the software is fast, if it is bug free and if they regularly update it with new features and functionality that makes your life better”.

APIs now are business models. You can now place a whole company - with hundreds, thousands of people - behind a set of APIs. APIs can very quickly be improved on in the background, served to many users and even get piped to other APIs.

Here are examples from Stripe and from Shopify (I’m not associated with them). These recipe books with sharp technical commands don't need UI updates, language translations or you to commute to a location for them to work. Each API section could be maintained by one or multiple teams, changed many times a day.

Examples:

Check your Stripe balance : GET /v1/balance

Get a list of your Shopify customers: GET /admin/api/2024-07/customers.json

To handle internet volume and shopper's little patience for slow experiences, we need standardisation - whatever doesn’t fit in the API pipe must stay behind, for better or for worse. When you use smoke signals to signal danger to your tribe, attacks to your army, or a new pope, the ones receiving the message already know what it means from a preset list of answers. That's how you scale communications with APIs.

Talking to other machines for fun and learnings

Time for a practical example. We’ll use your phone to get a custom notification using the same internet verbs that APIs use and adapt.

Download ntfy on your phone for this part. (I’m not associated with them) This will let you receive push notifications for free on your phone. It's free to use on public channels - for our purposes it's perfectly fine.

Think of a title for your channel. I’m picking chocolate_chip_pancake.

Whilst I’m working on embedding this, you can still try it in a very simple app I built here. Now make sure the title/topic is spelled the same way, and fire away! Sure enough, you will be able to receive your first ever self-made notification.

The code for the app is here.

!WARNING! The app is not really using an API call! APIs themselves simplify and hide other tools that enable their job. For this app, the more low level tool being used to send a request to ntfy is cURL.

cURL takes care of delivering the message efficiently and securely, then showing you the server's reply. Connected software everywhere makes use of these lower level tools to work. In this case, the ntfy server does the rest of the work of piping that message to your phone as a notification.

You can try this code yourself in a terminal:

curl -d "It’s pancake time! 🥞" ntfy.sh/chocolate_chip_pancake

-d specifies there is data to be sent. There is more discussion of ntfy here.

Going one layer down

Why do modern APIs look the way they look? Because they have evolved from a need for scale and security.

The most successful approach to designing them is REST. It stands for representational state transfer, which means there is a separation between the pancake notification you ask ntfy to send, the version of the message ntfy holds on their servers, and the one that shows up on your mobile phone. The message "It’s pancake time! 🥞" will be transcribed many times before arriving on your phone's notification center.

Why REST? Because REST promotes a cookie cutter approach to making API requests that is highly approachable no matter what programming language you decide on using. The formatting also uses very well known conventions, in REST's case the HTTP standard.

HTTP (Hypertext Transfer Protocol) is the grammar of choice for web-scale data format and transfer. The intention to use it for scale means HTTP doesn't just apply to websites! The cURL tool above is one way to transfer data using the HTTP protocol and a good way to test APIs manually.

What patterns do you see between Stripe’s and Shopify’s examples? As you look through the functionality, notice how they don’t promise the whole world, but very specific, targeted and regimented operations to help you use their service.

The HTTP actions used are linked to elemental data changes:

Operation

Action


The GET method requests a representation of the specified resource. Requests using GET should only retrieve data.


The POST method submits an entity to the specified resource, often causing a change in state or side effects on the server.


The PUT method replaces all current representations of the target resource with the request payload.


The DELETE method deletes the specified resource.

You can find 4-5 other less used actions here. They are called request methods because a 'server' like the ntfy server gets requests from another computer, like ours, which it acts on. They are also called verbs. Our computer started the discussion so it is called the client in this case, but we could make it listen to requests too and make it act as a server.

Designing REST APIs in very brief terms

APIs are can be judged on:

  • code instructions you use to operate the machine

  • the response the machine gives to instructions,

  • quality of human-focused documentation - so long as it is not a crutch for using the API

Skipping the very interesting background, API experiences can be judged by:

  • how self contained the information in each request and response is

  • how clear and predictable an instruction is

  • how much separation there is between the business logic the requester can use and the underlying data structures the responder is working with

  • how quality\satisfaction is perceived by users based on external attributes of the system

    • speed

    • power and precision of instructions available

    • usability of API server responses

There are maturity models for this, which have matured 🧐 over the years. Importantly, as raised in (Salvadori & Siqueira, 2015), all of these are subjective! Each API has different needs and goals. My product spidey sense is tingling and triggering a mantra that has become really 'loud' lately:

TALK TO YOUR F****** CUSTOMERS!

Let's get to the pretty models in a bit and take some learnings from a standout achievement of the British government: its Digital Service. These are their tips for API design:

  • Start with your customer's needs

  • Check for existing APIs and reuse or expand them if you can

  • Build your API before the system powering it so you prioritise user experience, API self sufficiency, and separation of business logic from internal system actions

  • Write out an API specification i.e. following OpenAPI

    • Test your assumptions with customers against the specification you wrote

  • Iterate on the design and decide what your live API will look like. It needs to be backwards compatible, so it's much easier to iterate on it before it is published.

Moving from there into the principles REST was designed to meet:

  • uniform interface – all API requests for the same resource should have the same uniform resource identifier (URI)

    • i.e. POST/ v1/ customers/ :id, GET/ v1/ customers/ :id and DELETE/ v1/ customers/ :id are all uniform calls on the 'customer' resource to create, get information on, and delete a customer with a specific id

      • they are self descriptive if you are familiar with HTTP verbs

      • they represent the customer resource, but they are not for example the actual database entry for the customer

      • the server responses should be uniform too. for example, the GET response should let the client know how to retrieve the full list of Stripe customers if you remove the id filter

  • client and server – these must be independent of each other and separated by a well defined interface

  • statelessness – this means the client knows everything needed to operate on the server in one request, and so does the server. This means more computers running the server software can be added behind an API to scale the service without having to share context previous client requests.

  • cacheable – resources should be cached on the client side and server side, to improve performance and scalability. Caching is when you store information you already know. This saves resources and optimises for scaling

  • layered system – allows for security and scaling services to be placed between the client and the server, to make the service more reliable and secure

Shiny maturity model time:


  • Level 0 - anything can be requested, you cannot predict how to use the API without extensive docs, doesn't qualify as RESTful

    • URI: /bookingService HTTP: POST Actions: retrieve destinations/hotels/rooms; add/cancel a reservation; etc.

  • Level 1 - general resources are introduced and can be grouped with common sense, it's still not restful

    • URI: /bookingReservations HTTP: POST Actions: add/cancel reservations

    • URI: /bookingDestinations HTTP: POST Actions: retrieve destinations

  • Level 2 - HTTP verbs added - further specialising the resource and narrowing down the outcome of each action, partially RESTful

    • URI: /reservations HTTP: GET Actions: get reservations according to certain criteria

    • URI: /reservations HTTP: POST Actions: add/cancel reservations

  • Level 3 - response includes links to other actions you can take on a resource, a bit like how the 'See Also' section on wikipedia works for humans. fully RESTful because these links should be everything the client needs to navigate the API without the need for documentation

    • URI: /room HTTP: GET Actions: get room information according to specific filtering

      • Response: { customerId: "1", reservations: [{room: "102", checkin: "10-11-2020", checkout: "11-14-2020", price: "100", href: "https://localhost:8080/room/102"}] }

        • Where href is a link to more information about room 102. This is useful because then the client code can be adapted to make more probing requests if needed. This can also be useful to help get more information about error responses.


  1. Identification of Resources

  2. Manipulation of Resources through Representations

  3. Self-Descriptive Messages

  4. Hypermedia as the Engine of Application State

With 4. the API provides information in a way that guides the client through the application. The client starts with a basic entry point (like a URL) and gets responses with links and instructions on what to do next. The client uses these instructions to decide how to interact with the service.

It also covers the impact of REST on functional properties:

  • Network Performance: Coarse grained, document oriented messages are encouraged.

  • Network Efficiency: Visibility enables caching which improves network efficiency.

  • Visibility: Messages are self-describing and can be understood by intermediaries. This is the foundation for caching and trust i.e. firewall admins can understand the messages and allow exactly the traffic they want to let through.

  • Evolvability: Coupling between client and server is removed, server owners need not know about client particularities to evolve the servers without breaking clients.

  • Extensibility: Increased by loose coupling.

  • Simplicity: Maximized by uniformity of connectors and identifiers

  • Scalability: Message self-descriptiveness constraint mandates message statelessness which in turn mandates the stateless server constraint. The latter guarantees scalibility.

  • Initial cost: Considerably high learning curve, cost of proper media type design, highest client side effort (burdon of loose coupling is placed on client side programming model)

  • Maintenance cost:Low due to simplicity, easy to find knowledgeable staff. (While it is considerable effort to design a proper media type, understanding it has a low entry barrier)

  • Evolution cost: REST optimizes for evolvability. Server owners need not consider impact of evolution on clients.


In a time where automated agents may become widely successful, this angle is very interesting. For day to day operations with this next generation of API, a software agent would not need to think anymore, only act directly on their goal. With initial exceptions, this approach may need only minima; error handling - all actions in the system are predetermined. This comes with the heavy price of having to think very hard through the domain about what flows should be standardised.

Even with the new proposed maturity dimensions of:

  • data: understandability and discoverability

  • profile: context and recommended domain-specific recipes

mapping a business outcome to a range of very intensive and very precise operations will still be very important for the value of the API.

Where were we?

Right, how to apply API design to working remotely.

Two things are apparent when you pick up the phone line to talk to another person - you probably wont be able to work faster, and its not really the same as being there.

When everyone is on a multiplex line, the plot thickens. You still can’t talk or chat over other people. Whatever communication is happening human to human, is done in a sequential and stateful way. And humans process information in a sequential way too.

If you apply some principles of API design, you can make yourself, your team and your professional network work more effectively. The trick is to standardise the right interactions which you can do at volume, and letting go of more ‘genuine’ dimensions. So maybe you can't duplicate yourself, but you can use documents, shape how information flows to your team, and shape your organisation itself so remote work can be more effective.

It also seems not all work needs to be optimised this way. Solo working is mostly fine. But colaboration is where this is more important.

The point of APIs is that while they can be complex and have many functions, they should not be complicated to use. An API that takes a long time to implement is not a great API. There will always be a competitor around the corner with better user experience.

Making remote work more deliberate

With that in mind, here are some API design rules of thumb that have interesting parallels to repetitive work and working remotely in a distributed way. Could you collaborate effectively with another human in a completely different timezone?

Planning - API planning planning is always the most important step.

  • Do you know what your goal is? How often do you review it?

    • This year?

    • This month?

    • Today?

  • Do you know what kind of work you will not do? Or delegate?

    • Are your boundaries well defined?

  • Are there any upcoming events that will affect your capacity to help effectively?

Clear communication - API 'contracts' clearly define how they can be used and don't hide inconsistent behavior

  • How can you communicate more clearly?

  • How can you communicate more regularly?

  • How connected and informed do your teammates feel?

Collaboration and support - APIs often require collaboration between different teams or organisations to develop, maintain, and support them.

  • Do you know your teammate’s goals?

  • Do you know their preferred way of collaborating?

  • Do you know how best to contribute to shared goals?

Documentation - A well-designed API must have clear, concise, and easily understood documentation to enable developers to integrate with it effectively.

  • Do you have a guide to how best to deal with yourself?

    • Managers in tech often create ‘readme’ files, but you don’t have to be one to create yours. Some examples here

  • Is your self documentation, your team’s documentation, and your organization’s documentation clear?

    • Do you have guidelines?

  • Do you keep your documentation fresh?

Standardization - Consistent API design patterns, naming conventions, and error handling make it easier for developers to understand and use the API.

  • How well do you know your tools of the trade?

    • Are you aware of how you can update them in the future?

  • Are you comfortable with all of them?

  • Do you apply your tools, methods and processes consistently?

Scalability and performance - Good API design accounts for scalability and performance, ensuring that the API can handle growing numbers of users and data without degradation in quality.

  • How can you scale a team and maintain performance?

  • What are the tradeoffs there?

  • How should you set goals for teams to promote collaboration?

  • What parts of your work are repetitive? Which ones may no longer be needed?

Work empathetically and testing - A well-designed API should be easy to use and understand, minimising developer friction and promoting adoption.

  • What assumptions am I making by communicating or acting this way?

  • What happened the last time someone was confused with something I said?

  • When you are communicating a new concept, method or to a new audience, how often do you test it first?

Flexibility and adaptability - A successful API should be able to evolve and adapt to changing requirements and technology trends without disrupting existing users.

  • When you introduce a new process, what do you keep the same?

    • How do you plan for change?

  • Are there existing ‘building blocks’ you could keep the same, instead of redoing it from scratch?

  • Are your processes and communication fault tolerant? Do they forgive genuine or systemic mistakes like cognitive biases?

Leverage your network! APIs are more powerful when they are connected together

  • Do you know what skills there are on your network?

  • How often do you leverage your network?

  • Does your network serve a purpose?

  • Do you see potential links in your network you could make happen?

And lastly, know when doing something in person would be more effective, if you have that luxury!
